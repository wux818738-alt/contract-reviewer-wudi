# 合同审核方法论

本文档详细说明合同审核的具体方法和实践指南。

---

## 一、审核前的准备工作

### 1.1 了解交易背景

在开始审核前，需要了解：

- 合同主体的基本情况（注册资本、经营范围、信用状况）
- 交易标的和交易目的
- 交易结构（一次性交付/分批交付/框架协议）
- 交易金额和付款安排
- 履约期限和关键节点
- 是否有特殊监管要求

### 1.2 确定审核立场

| 立场 | 审核重点 | 典型策略 |
|------|----------|----------|
| **甲方（付款方）** | 付款条件、验收标准、违约责任上限 | 限制付款触发条件，争取验收主动权 |
| **乙方（收款方）** | 付款保障、知识产权、违约责任触发条件 | 明确付款时间和方式，限制违约责任范围 |
| **中立立场** | 均衡保护、条款平衡 | 标注双方权利义务是否对等 |

### 1.3 收集参考资料

- 优秀合同模板（行业通行版本）
- 客户提供的参考合同
- 相关法律法规和司法解释
- 行业监管要求
- 类似交易的合同版本

## 二、合同结构识别

### 2.1 常见合同结构

```
封面
目录（可选）
正文
  ├── 定义条款
  ├── 标的条款
  ├── 价款/报酬条款
  ├── 履行条款
  ├── 验收条款
  ├── 知识产权条款
  ├── 保密条款
  ├── 违约责任条款
  ├── 解除终止条款
  ├── 争议解决条款
  └── 其他条款（通知、生效等）
签署页
附件
```

### 2.2 条款编号识别

常见编号格式：
- 第X条/第X款
- X. / X.X / X.X.X
- (X) / (X.X)
- 一、/ （一）/ 1.

使用结构解析引擎可自动识别：

```bash
python3 scripts/contract_parser.py 合同.docx 结构.json
```

### 2.3 定义词识别

定义词通常出现在：
- 首部定义条款
- 条款中的"XX是指……"
- 特定用语的首字母大写

检查要点：
- 定义词是否在全文统一使用
- 定义词是否被使用
- 是否有循环定义
- 是否有定义冲突

## 三、审核维度详解

### 3.1 准确性审核

**审核目标**：确保条款表意唯一确定，无歧义。

**常见问题**：

| 问题类型 | 示例 | 修正方案 |
|----------|------|----------|
| 时间不明确 | "合理期限内" | "收到通知后 15 个工作日内" |
| 金额不确定 | "违约金按实际损失计算" | "违约金为合同总金额的 10%" |
| 主体不明确 | "相关方" | "甲方指定的第三方，须提前书面通知乙方" |
| 标准不具体 | "符合行业标准" | "符合 GB/T XXXXX-20XX 标准" |
| 范围不清晰 | "相关费用" | "包括但不限于运输费、保险费、安装调试费" |

**检查清单**：

- [ ] 是否存在"合理""适当""相关"等模糊用语
- [ ] 时间节点是否有明确起算点
- [ ] 金额是否有明确计算方式
- [ ] 履行标准是否有客观依据
- [ ] 权利义务是否有明确主体

### 3.2 逻辑性审核

**审核目标**：确保条款间协调一致，无冲突。

**常见问题**：

| 问题类型 | 示例 | 修正方案 |
|----------|------|----------|
| 时间冲突 | 付款期限早于验收期限 | 调整时间顺序 |
| 权利冲突 | 单方解除权与自动续期矛盾 | 明确优先适用顺序 |
| 责任冲突 | 违约金上限低于定金罚则 | 统一责任计算方式 |
| 程序冲突 | 争议解决条款约定仲裁又约定诉讼 | 选择其一 |
| 条件缺失 | 只有违约金，无违约行为定义 | 明确违约情形 |

**检查清单**：

- [ ] 时间轴是否连贯
- [ ] 权利义务是否对应
- [ ] 违约责任是否有触发条件
- [ ] 解除条件是否可执行
- [ ] 争议解决是否唯一确定

### 3.3 规范性审核

**审核目标**：确保条款符合行业惯例和法律规范。

**常见问题**：

| 问题类型 | 示例 | 修正方案 |
|----------|------|----------|
| 格式条款问题 | 未提示免责条款 | 加粗提示并留出阅读时间 |
| 定金问题 | 定金超过合同总价 20% | 调整至法定上限内 |
| 违约金问题 | 违约金显著过高或过低 | 调整至实际损失的 30% 以内 |
| 争议解决问题 | 约定无管辖权的法院 | 约定合同履行地或被告住所地法院 |
| 强制性问题 | 约定排除强制性法律规定 | 删除或修改 |

**检查清单**：

- [ ] 是否违反法律强制性规定
- [ ] 是否符合行业通行做法
- [ ] 格式条款是否履行提示义务
- [ ] 定金、违约金是否在合理范围
- [ ] 管辖约定是否有效

### 3.4 简洁性审核

**审核目标**：删除冗余表达，提高可读性。

**常见问题**：

| 问题类型 | 示例 | 修正方案 |
|----------|------|----------|
| 同义反复 | "违约金/赔偿金/补偿金" | 统一为"违约金" |
| 冗余限定 | "在本合同有效期内，在合同履行过程中" | "合同履行期间" |
| 重复定义 | 同一定义在多处出现 | 集中到定义条款 |
| 过度引用 | "详见第 X 条第 X 款第 X 项" | 引用条款编号即可 |
| 无效条款 | "按法律法规执行" | 删除（本应执行） |

**检查清单**：

- [ ] 是否存在重复表述
- [ ] 是否存在无效条款
- [ ] 是否存在过度限定
- [ ] 定义词是否集中定义
- [ ] 引用是否必要

### 3.5 流畅性审核

**审核目标**：确保行文通顺，易于理解。

**常见问题**：

| 问题类型 | 示例 | 修正方案 |
|----------|------|----------|
| 长句嵌套 | 超过 50 字无标点 | 拆分为多个短句 |
| 主语缺失 | 多个条款连续出现时 | 补充主语或使用条款引用 |
| 被动语态过多 | "应被支付""应被验收" | 使用主动语态 |
| 中英文混用 | "Party A 应当 payment" | 统一使用中文 |

**检查清单**：

- [ ] 句子长度是否适中
- [ ] 主语是否清晰
- [ ] 是否使用主动语态
- [ ] 语言风格是否统一

### 3.6 工整性审核

**审核目标**：确保格式统一，排版规范。

**常见问题**：

| 问题类型 | 示例 | 修正方案 |
|----------|------|----------|
| 编号不统一 | "1."与"（1）"混用 | 统一编号格式 |
| 标点不统一 | 中英文标点混用 | 统一使用中文标点 |
| 字体不统一 | 正文多种字体混用 | 统一字体（如宋体） |
| 行距不统一 | 部分段落行距不一致 | 统一行距（如 1.5 倍） |
| 缩进不统一 | 首行缩进不一致 | 统一缩进（如 2 字符） |

**检查清单**：

- [ ] 编号格式是否统一
- [ ] 标点符号是否统一
- [ ] 字体字号是否统一
- [ ] 行距段距是否统一
- [ ] 缩进是否统一

## 四、风险等级判定

### 4.1 高风险判定标准

满足以下条件之一的，判定为高风险：

- 可能导致合同无效或部分无效
- 可能导致重大经济损失（超过合同金额 30%）
- 可能导致无法主张主要权利
- 可能导致承担无限责任
- 可能导致核心商业秘密泄露
- 违反法律强制性规定

**必须修改**，并在批注中说明法律后果。

### 4.2 中风险判定标准

满足以下条件之一的，判定为中风险：

- 可能导致合同解释争议
- 可能导致次要权利受损
- 可能承担不合理责任
- 可能丧失关键抗辩权
- 存在明显不对等条款

**建议修改**，并在批注中提供备选方案。

### 4.3 低风险判定标准

满足以下条件之一的，判定为低风险：

- 表述不当但不影响实质权利
- 格式不规范但不影响阅读理解
- 存在冗余表达但不造成歧义
- 约定不完整但可依法补充

**可选修改**，在报告中列出即可。

## 五、自动条款推荐（Clause Library）

### 5.0 条款库工作原理

在审核过程中，每当识别出风险，立即调用条款推荐引擎：

```
风险识别
  ↓
调用 clause_recommender.py
  ├─ 输入：合同类型 + 风险关键词 + 合同关键词
  ↓
引擎返回匹配条款（含甲方/乙方立场建议）
  ↓
在批注中附加 suggestion 字段
  ↓
同时在 changes.json 中记录 clause_source
```

**AI 审核时的标准工作流（自动执行，无需人工干预）：**

```bash
# 1. 自动检测合同类型
python3 scripts/contract_parser.py 合同.docx 结构.json --detect-type
# → 返回 confidence_top_type，如 "house-sale"

# 2. 识别风险后，调用推荐引擎
python3 scripts/clause_recommender.py \
  --type house-sale \
  --risks "违约金偏高" "收款账户空白" \
  --top-k 2
# → 返回最匹配的替代条款

# 3. 将推荐条款写入 changes.json
```

### 5.1 条款库文件结构

所有条款位于 `references/clause-library/`：

```
clause-library/
├── payment/
│   ├── prepaid-30pct.json     # 预付30%货款
│   ├── milestone.json          # 里程碑付款
│   ├── net-30.json            # 月结30天
│   ├── escrow.json             # 资金监管（房屋买卖）
│   └── escrow-construction.json
├── penalty/
│   ├── daily-5-per-10k.json   # 逾期每日万分之五（★推荐）
│   ├── 20pct-fixed.json       # 固定比例20%
│   └── labor-daily-3pct.json  # 劳动合同滞纳金
├── dispute/
│   ├── court-shanghai.json     # 原告所在地法院
│   ├── court-specified.json    # 指定法院
│   ├── arbitration-cietac.json # CIETAC仲裁（★推荐涉外）
│   └── arbitration-local.json  # 国内仲裁（★推荐国内）
├── force-majeure/
│   └── standard.json           # 标准不可抗力
├── ip/
│   ├── work-for-hire.json     # 成果归甲方
│   ├── joint-own.json         # 共有知识产权
│   └── license-back.json      # 反向许可
├── delivery/
│   ├── acceptance-7days.json  # 7天验收+逾期视为合格
│   └── third-party-acceptance.json
├── confidentiality/
│   ├── nda-3years.json        # 3年保密
│   └── nda-perpetual.json     # 永久保密（限核心）
├── termination/
│   ├── convenience-30days.json # 随时解除
│   └── breach-cure-15days.json # 违约补救期15天
└── general/

    ├── governing-law-mainland.json
    └── entire-agreement.json

### 5.2 风险 → 条款自动映射

AI 在审核时，应根据识别的风险自动调用相应类别的条款：

| 识别的风险 | 推荐条款类别 | 首选推荐 | 原因 |
|-----------|------------|---------|------|
| 违约金无上限 / 偏高 | penalty | daily-5-per-10k | 日万分之一≈年化3.65%，合理且可主张 |
| 收款账户空白 / 未指定 | payment / escrow | escrow | 资金监管直接解决账户风险 |
| 知识产权归属不明 | ip | work-for-hire | 明确归委托方 |
| 无不可抗力条款 | force-majeure | standard | 标准版本，覆盖完整 |
| 无保密条款 / 期限缺失 | confidentiality | nda-3years | 3年是常见标准 |
| 解除权不对等 / 无补救期 | termination | breach-cure-15days | 15天补救期缓解轻微违约风险 |
| 无争议解决条款 | dispute | court-specified | 指定己方所在地法院 |
| 变更无书面要求 | general | amendment-written | 防止口头承诺争议 |
| 验收无期限 | delivery | acceptance-7days | 逾期视为验收，防止拖延 |
| 付款无保障 | payment | milestone | 分阶段付款降低风险 |

### 5.3 推荐条款的使用规范

在批注中写入 `suggestion` 字段时，**必须遵守以下规范**：

**规范 1：必须保留方括号 `[___]` 占位符**
```
❌ 错误：违约金为合同总价的 5%
✅ 正确：违约金为合同总价的【20】%
```

**规范 2：必须标注立场来源**
```
在 suggestion 末尾注明：
来源：clause-library/penalty/daily-5-per-10k.json（甲方/乙方立场版）
```

**规范 3：必须注明法律依据**
```
法律依据：《民法典》第585条
```

**规范 4：对于房屋买卖合同，优先推荐资金监管条款**
```
收款方风险：务必添加资金监管（clause-library/payment/escrow.json）
付款方风险：务必删除预付款比例或降低至20%
```

## 六、批注撰写规范

### 6.1 批注结构

每条批注应包含以下要素：

```
【问题】
[描述具体问题]

【风险】
[说明为什么有风险，可能导致什么后果]

【建议】
[提出具体修改建议]

【参考条款】
[提供可替换的完整条款文本]
```

### 5.2 批注示例

**示例 1：高风险批注**

```
【问题】
第 5.2 条约定"任何情况下违约金总额不超过合同总金额的 5%"，
低于实际损失可能无法获得充分赔偿。

【风险】
若实际损失超过合同金额 5%，守约方无法就超出部分主张赔偿，
可能承担重大损失。

【建议】
删除违约金上限条款，或提高至合同金额的 30%。

【参考条款】
"违约金不足以弥补实际损失的，违约方应补足差额。"
```

**示例 2：中风险批注**

```
【问题】
第 7 条仅约定诉讼管辖，未约定仲裁选项。

【风险】
诉讼程序较长，且二审终审；仲裁一裁终局，
更有利于快速解决争议。

【建议】
增加仲裁条款，或约定仲裁与诉讼可选择。

【参考条款】
"因本合同引起的争议，双方协商解决；协商不成的，
任何一方均可向 XX 仲裁委员会申请仲裁，或向合同履行地人民法院提起诉讼。"
```

### 5.3 批注要求

- 必须具体，不得泛泛而谈
- 必须提供修改建议
- 必须提供参考条款
- 必须说明风险原因
- 高风险必须说明法律后果
- **高风险**批注必须提供 `reason`（修改原因）和 `legal_basis`（法律依据）
- **中风险**批注建议提供 `reason` 和 `legal_basis`

### 5.4 changes.json 批注字段规范

每条批注在 changes.json 中应包含以下字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| `comment` | 是 | 问题分析（简短描述问题是什么） |
| `severity` | 是 | 高风险 / 中风险 / 低风险 |
| `reason` | 建议 | 修改原因（法理分析，为什么需要修改） |
| `legal_basis` | 建议 | 法律依据（具体到条文，格式：《法律名》第X条） |
| `suggestion` | 建议 | 建议条款（完整的可直接使用的替换文本） |
| `highlight_text` | 否 | 高亮文本（精确匹配，用于定位） |
| `category` | 否 | 条款类别（用于报告统计） |

**字段说明：**

- `comment`：描述问题本身，客观中立。如"该违约金条款约定过高"
- `reason`：解释为什么要修改，从法理和商业角度分析。如"未约定违约金上限，根据《民法典》第585条第2款，约定的违约金过分高于造成的损失的，法院可依申请予以适当减少"
- `legal_basis`：引用具体法律条文，用分号分隔多个条文。如：《民法典》第585条第2款；参照《最高人民法院关于适用〈中华人民共和国民法典〉合同编通则若干问题的解释》第65条
- `suggestion`：给出可直接使用的完整替换条款，不仅仅是方向性建议

**法律依据参考：**

详见 `references/legal-basis.md`，其中包含常用合同问题的法律依据映射。

**完整示例：**

```json
{
  "paragraph_index": 12,
  "highlight_text": "违约金为合同总金额的30%",
  "comment": "该违约金约定过高，可能被认定为"过分高于造成的损失"",
  "severity": "高风险",
  "reason": "未约定违约金上限，根据《民法典》第585条第2款，约定的违约金过分高于造成的损失的，法院可依申请予以适当减少；参照合同编通则解释第65条，一般以超过损失30%为认定标准",
  "legal_basis": "《民法典》第585条第2款；参照《最高人民法院关于适用〈中华人民共和国民法典〉合同编通则若干问题的解释》第65条",
  "suggestion": "违约金为合同总金额的20%；因一方违约给另一方造成实际损失超过该违约金的，违约方还应赔偿超出部分的损失，但赔偿总额不超过合同总金额的30%"
}

## 七、修订建议撰写

### 6.1 修订原则

1. **最小修改** - 只改有问题的部分
2. **保留原意** - 不改变当事人真实意思
3. **语言一致** - 保持原有表述风格
4. **格式统一** - 不改变原有排版

### 6.2 修订示例

**原文**：
> 乙方应在收到甲方通知后合理期限内完成修复。

**修订为**：
> 乙方应在收到甲方书面通知后 10 个工作日内完成修复。

**修订理由**：
"合理期限"缺乏明确标准，可能导致争议。约定具体期限有利于执行。

## 八、审查报告撰写

### 7.1 报告结构

```
一、合同概况
二、风险问题清单（按风险等级排序）
三、重点修改内容
四、格式及一致性问题
五、需确认事项
六、统计数据
```

### 7.2 报告要求

- 语言简洁，重点突出
- 风险问题按等级排序
- 每个问题说明条款位置
- 统计数据准确完整
- 需确认事项单独列出

## 九、多轮迭代管理

### 10.1 迭代流程

```
第一轮：初次审核
  ├── 解析结构
  ├── 识别风险
  ├── 生成修订批注版
  ├── 生成清洁版
  └── 归档到 round-1/

第二轮：复审
  ├── 对比上一轮清洁版
  ├── 识别新增问题
  ├── 生成修订批注版
  └── 归档到 round-2/

第 N 轮：定稿
  ├── 最终确认
  ├── 生成最终清洁版
  └── 标记为 final/
```

### 10.2 版本对比重点

- 新增条款是否有新风险
- 修改后条款是否解决原问题
- 是否引入新的不一致
- 定稿版本是否可执行

## 十、附表

### 10.1 常见风险条款速查

| 条款类型 | 高风险情形 | 审查要点 |
|----------|------------|----------|
| 定金条款 | 超过 20% | 是否超过法定上限 |
| 违约金条款 | 无上限或上限过低 | 是否合理（实际损失 30% 以内） |
| 解除条款 | 单方解除权不对等 | 是否约定解除条件、通知期限 |
| 争议解决 | 约定无效管辖 | 是否属于专属管辖范围 |
| 知识产权 | 权利归属不明 | 是否明确约定权利归属 |
| 保密条款 | 无保密期限 | 是否约定保密期限和范围 |
| 责任限制 | 单方免责 | 是否合理限制责任范围 |

### 10.2 条款审查顺序

1. 定义条款（确定术语含义）
2. 标的条款（确定交易内容）
3. 价款条款（确定金额和时间）
4. 履行条款（确定方式和期限）
5. 违约条款（确定责任承担）
6. 解除条款（确定终止条件）
7. 争议解决条款（确定争议处理方式）
8. 其他条款（通知、生效等）
