# 输出文件规范

## 一、默认输出文件

### 必需文件

| 文件名 | 格式 | 说明 |
|--------|------|------|
| 合同名-修订批注版.docx | Word | 含 Track Changes 和 Comments |
| 合同名-清洁版.docx | Word | 接受所有修订后的版本 |
| 合同名-审查报告.txt | 文本 | 风险总结和修改建议 |
| 合同名-changes.json | JSON | 结构化变更记录 |

### 可选文件

| 文件名 | 格式 | 说明 |
|--------|------|------|
| 合同名-结构解析.json | JSON | 合同结构解析结果 |
| 合同名-对比报告.txt | 文本 | 多轮迭代对比（如有） |

---

## 二、目录结构

### 单轮审核

```
合同名-Output/
├── 合同名-修订批注版.docx
├── 合同名-清洁版.docx
├── 合同名-审查报告.txt
├── 合同名-changes.json
└── 合同名-结构解析.json
```

### 多轮迭代

```
合同名-Output/
├── manifest.json              # 迭代管理清单
├── 合同名-结构解析.json       # 全局结构文件
├── round-1/                   # 第一轮
│   ├── 合同名-修订批注版.docx
│   ├── 合同名-清洁版.docx
│   ├── 合同名-审查报告.txt
│   └── changes.json
├── round-2/                   # 第二轮
│   ├── ...
└── final/                    # 最终定稿（可选）
    └── 合同名-定稿版.docx
```

## 三、changes.json 格式

### 基本结构

```json
{
  "round": 1,
  "author": "Claude",
  "date": "2026-04-15T12:00:00Z",
  "source_file": "合同名.docx",
  "revisions": [...],
  "comments": [...],
  "summary": {...}
}
```

### revisions 字段

```json
"revisions": [
  {
    "id": "rev_001",
    "paragraph_index": 42,
    "clause_number": "第五条",
    "original_text": "违约金为合同总金额的 30%",
    "revised_text": "违约金为合同总金额的 20%",
    "reason": "30% 可能被认定为过高",
    "risk_level": "中风险",
    "category": "违约责任"
  }
]
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 否 | 修订唯一标识 |
| paragraph_index | int | 是 | 段落索引（0-based） |
| clause_number | string | 否 | 条款编号 |
| original_text | string | 是 | 原文 |
| revised_text | string | 是 | 修订文 |
| reason | string | 否 | 修订原因 |
| risk_level | string | 否 | 风险等级 |
| category | string | 否 | 条款类别 |

### comments 字段

```json
"comments": [
  {
    "id": "cmt_001",
    "paragraph_index": 55,
    "clause_number": "第七条",
    "highlight_text": "争议解决",
    "comment": "建议增加仲裁选项",
    "severity": "中风险",
    "reason": "仅约定诉讼管辖，未提供仲裁选项，限制了当事人的争议解决方式选择权",
    "legal_basis": "《仲裁法》第16条；参照《最高人民法院关于适用〈中华人民共和国民事诉讼法〉的解释》",
    "suggestion": "建议增加仲裁选项："甲乙双方约定争议可提交XX仲裁委员会仲裁，或向有管辖权的人民法院提起诉讼"",
    "category": "争议解决"
  }
]
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 否 | 批注唯一标识 |
| paragraph_index | int | 是 | 段落索引 |
| clause_number | string | 否 | 条款编号 |
| highlight_text | string | 否 | 高亮文本 |
| comment | string | 是 | 批注内容（问题分析） |
| severity | string | 否 | 严重程度（高风险/中风险/低风险） |
| reason | string | 否 | 修改原因 |
| legal_basis | string | 否 | 法律依据（如《民法典》第585条第2款） |
| suggestion | string | 否 | 建议条款 |
| category | string | 否 | 条款类别 |

### summary 字段

```json
"summary": {
  "total_revisions": 15,
  "total_comments": 8,
  "high_risk": 3,
  "medium_risk": 5,
  "low_risk": 7,
  "categories": {
    "违约责任": 5,
    "争议解决": 2,
    "知识产权": 3,
    "保密": 2,
    "其他": 3
  }
}
```

## 五、manifest.json 格式

### 基本结构

```json
{
  "project_name": "合同名",
  "created_at": "2026-04-15T10:00:00Z",
  "current_round": 2,
  "rounds": [
    {
      "round": 1,
      "created_at": "2026-04-15T10:00:00Z",
      "files": {...},
      "summary": {...}
    }
  ]
}
```

### rounds[].files 字段

```json
"files": {
  "revised": "round-1/合同名-修订批注版.docx",
  "clean": "round-1/合同名-清洁版.docx",
  "report": "round-1/合同名-审查报告.txt",
  "changes": "round-1/changes.json"
}
```

### rounds[].summary 字段

```json
"summary": {
  "revisions_count": 15,
  "comments_count": 8,
  "high_risk": 3,
  "medium_risk": 5,
  "low_risk": 7
}
```

## 六、文件命名规则

### 单轮审核

| 文件类型 | 命名格式 | 示例 |
|----------|----------|------|
| 修订批注版 | {合同名}-修订批注版.docx | 采购合同-修订批注版.docx |
| 清洁版 | {合同名}-清洁版.docx | 采购合同-清洁版.docx |
| 审查报告 | {合同名}-审查报告.txt | 采购合同-审查报告.txt |
| 变更记录 | {合同名}-changes.json | 采购合同-changes.json |

### 多轮迭代

| 文件类型 | 命名格式 | 示例 |
|----------|----------|------|
| 第 N 轮修订版 | {合同名}-修订批注版-R{N}.docx | 采购合同-修订批注版-R2.docx |
| 第 N 轮清洁版 | {合同名}-清洁版-R{N}.docx | 采购合同-清洁版-R2.docx |
| 第 N 轮报告 | {合同名}-审查报告-R{N}.txt | 采购合同-审查报告-R2.txt |
| 第 N 轮变更 | changes.json | changes.json |

## 七、编码规范

### 文本文件

- 编码：UTF-8（无 BOM）
- 换行符：LF（Unix 风格）
- 缩进：2 空格（JSON），4 空格（Python）

### Word 文件

- 格式：.docx（Office Open XML）
- 兼容性：Word 2010 及以上
- 修订痕迹：w:ins/w:del 元素
- 批注：comments.xml + 标记

## 八、版本控制

### 版本号规则

- 主版本号：重大功能变更
- 次版本号：新增功能或引擎
- 修订号：Bug 修复

示例：`2.1.3` 表示第 2 主版本，第 1 次新增功能，第 3 次修复

### 变更日志

每次更新应记录在 CHANGELOG.md：

```markdown
## [2.0.0] - 2026-04-15

- 结构解析引擎 contract_parser.py
- 修订痕迹引擎 apply_changes.py

- 核心审核方法论基于法律审核通用实践
```
